---- 14. SECURITY ARCHITECTURE ----
API key (Anthropic):
  Stored in expo-secure-store (iOS Keychain / Android
  Keystore, hardware-backed). Key: ANTHROPIC_API_KEY.
  Format validation via validateKeyFormat() — detects
  and rejects Claude Pro OAuth tokens (sk-ant-oat*) with
  specific error message. Live-test via minimal Haiku
  ping. BYOK model: key never leaves device except in
  x-api-key header to api.anthropic.com over HTTPS.

PIN storage (authService.ts):
  SHA-256 of `dub_ai_pin_salt_${pin}`.
  *** FINDING: STATIC SALT. ***
  The salt string "dub_ai_pin_salt_" is constant and
  visible in source. Per-user random salt with storage in
  SecureStore is the expected pattern. For a 4-digit PIN
  (10,000-entry keyspace) a device-seized attacker can
  exhaustively pre-compute the SHA-256 in milliseconds.
  Mitigations in place: (a) PIN hash stored in
  SecureStore (Keychain) which is itself hardware-bound;
  (b) 5-attempt lockout with 30s cooldown (AuthGate.tsx
  MAX_PIN_ATTEMPTS / LOCKOUT_SECONDS). So the static salt
  is a defense-in-depth weakness, not an immediate
  compromise path.
  -> HIGH severity (90-day fix), NOT critical.

Biometric gating:
  authenticateBiometric() calls LocalAuthentication with
  disableDeviceFallback: true — prevents device-passcode
  fallback which would lower the security bar to the
  device's own screen-lock strength. Correct.

Secrets in codebase:
  grep for `password|secret|apikey|api_key|token` returned
  113 matches. Manual review: all are variable names,
  type fields, UI labels (e.g., "API Key"), third-party
  OAuth token types (FatSecret, Strava), or helper
  function names. ZERO hardcoded secrets detected.
  Strava:  src/config/strava.ts is a documented
           placeholder — clientId/secret intentionally
           stubbed pending production OAuth app.
  FatSecret: OAuth flow; credentials loaded from a
           placeholder module.

Other security controls:
  - PrivacyScreen overlay on AppState background/inactive
    (app/_layout.tsx:80-108) — screenshot-in-app-switcher
    protection, defaults ON.
  - Triple-tap QuickHide gesture (privacy overlay).
  - Indirect injection defense (see Phase 2 item 11).
  - Image EXIF stripping on every photo uploaded to
    Coach, food-scan, activity-logger
    (src/utils/imagePrivacy.ts, used in 6 call sites).
  - iCloud backup exclusion on AsyncStorage SQLite DB
    via custom Expo config plugin
    (plugins/exclude-from-backup.js) — sets
    NSURLIsExcludedFromBackupKey in AppDelegate.
  - ErrorBoundary at root layout catches render errors.

Encryption at rest (app-layer):
  src/utils/encryption.ts provides AES-256-CBC + PBKDF2
  (100k iterations, 256-bit key, 32-byte salt) via
  react-native-aes-crypto. *** IMPORTANT: Zero call sites.
  *** Import grep returned no consumers. Health data is
  therefore stored in AsyncStorage in plaintext JSON,
  relying on (a) iOS NSFileProtection (device-lock-bound
  disk encryption) and (b) the app-lock / biometric gate.
  Android has equivalent full-disk encryption on modern
  devices. For non-regulated consumer wellness apps this
  is defensible. For HIPAA-covered deployment (e.g.,
  partnering with a clinical provider) app-layer
  encryption should be activated.
  -> HIGH severity for HIPAA expansion pathway;
     MEDIUM for current direct-to-consumer use.
